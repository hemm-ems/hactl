# Contributing to hactl

Thanks for taking the time. This project has a short, opinionated stance on contributions —
please read it before opening anything.

## The deal

**We live in an AI world. If your AI can find a bug, it can fix it.** So if you (or your agent)
found a real problem, the most useful thing you can do is open a **pull request** with the fix.

- **AI-written issues are rejected.** Issues that are clearly generated by an LLM get closed as
  invalid (`ai-generated`). No exceptions, no hard feelings.
- **Human-written issues are always welcome.** If you hit something and can't fix it yourself,
  write it up *in your own words* — short and concrete beats long and polished.
- The bug forms ask for exact versions, the exact command, and the full output. That's not
  bureaucracy: it's what lets *my* LLM reproduce the problem and fix it. An issue I can't
  reproduce is an issue I can't fix.

Why this rule? AI generates plausible-sounding bug reports faster than any human can triage
them, and most of them are wrong, duplicated, or unreproducible. Pasting AI output into the
tracker moves work onto the maintainer. Pasting AI output into a *PR* moves work toward a fix.
Same tool, opposite direction.

## Found a bug?

1. **Can your AI fix it?** Open a PR. See below.
2. **Can't fix it?** Open a [bug report](https://github.com/hemm-ems/hactl/issues/new/choose) —
   written by you, by hand. Include:
   - `hactl version`
   - the relevant part of `hactl health`
   - the **exact** command you ran
   - what you expected vs. what happened (full output, including the `[~N tok]` line)
   - how you installed it (Homebrew / `go install` / binary / source)

## Have an idea?

Open a [feature request](https://github.com/hemm-ems/hactl/issues/new/choose) — again, written
by a human. Tell me the problem you're trying to solve, not just the solution. Even better:
prototype it and open a PR.

## Questions / how-to

Not a bug? Use [Discussions](https://github.com/hemm-ems/hactl/discussions). Most "how do I…"
answers are already in the [manual](docs/manual.md) (`hactl rtfm`).

## Pull requests

**AI-assisted PRs are welcome — that's the whole point.** I don't care whether a human or an
agent wrote the diff; I care that it's correct, tested, and reviewable.

Before you open a PR:

- [ ] It's linked to an issue, or the description explains the change and why.
- [ ] `make test-int` passes (Docker must be running first — see [AGENTS.md](AGENTS.md)).
- [ ] `golangci-lint run` is clean.
- [ ] Docs (`docs/manual.md`, README) updated if behavior changed.

Working on the code? [AGENTS.md](AGENTS.md) has the norms (tests, plan-before-acting,
read-before-writing, dry-run-by-default).

## Security

Never open a public issue for a vulnerability. Use
[private reporting](https://github.com/hemm-ems/hactl/security/advisories/new). See
[SECURITY.md](SECURITY.md).

## Code of Conduct

This project follows the [Contributor Covenant](CODE_OF_CONDUCT.md). Be decent.
